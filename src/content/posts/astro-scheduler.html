Shipping ASTRO's Scheduler

Every agentic security stack I've shipped so far has had the same failure mode: it only runs when a human remembers to run it. ASTRO's scheduler exists to remove that dependency entirely.

The shape of it

A Schedule table holds timezone-aware run times per stack. A poller wakes on an interval, claims due schedules with a Redis lock so two workers never double-fire, and hands off to a generic run_scheduled_stack task that reuses the same storage_consumer layer every manual run already goes through.

The goal was zero new execution paths — scheduled runs should be indistinguishable from manual ones by the time they hit the stack.
SCHEDULE.next_run_at <= now()
  → acquire_lock(schedule.id)
  → run_scheduled_stack.delay(schedule.stack_id)
  → release_lock(schedule.id)

Next up: surfacing drift between scheduled runs in a diff view, so a stack quietly changing behavior over time is something you notice instead of something you find out about later.
